## 2. sequence_service가 **반드시 공장 PC에서만** 돌아야 하는 이유

### 하드웨어 제약
| 의존 대상 | 연결 방식 | 원격 실행 가능? |
|----------|-----------|---------------|
| Ajin IO 보드 (AXL.dll) | PCIe/USB 물리 연결 | ❌ Windows + 물리 연결 필수 |
| HCR-10L 로봇 (Modbus TCP) | 공장 LAN (`127.0.0.1:9100` 기본) | △ 네트워크 도달만 되면 가능하나 지연/안전성 문제 |
| Vision TCP (`127.0.0.1:9200`) | 공장 LAN (비전 PC) | △ 동일 |
| PreFormServer (`10.145.113.3:44388`) | VPN | ✅ 가능 |
| MySQL (시퀀스 로그) | 공장 PC 로컬 | △ 네트워크로 열면 가능 |

**결론**: Ajin IO가 Windows DLL + 물리 하드웨어 의존이라 공장 PC를 벗어날 수 없음.

### 코드 구조 확인
- `sequence_service/app/main.py`: `venv\Scripts\python.exe` 같은 Windows 경로 기본
- `sequence_service/app/io/axl.py`: `from ctypes import WinDLL` — Windows 전용
- `SIMUL_MODE=true` / `AJIN_SIMULATION=true`로 **시뮬레이션은 어느 OS에서도 가능** (단, 실제 로봇은 안 움직임)

## 3. 운영 시나리오 3가지 — 장단점 비교

### 🅰️ 시나리오 A: 공장 PC 단독 실행 (현재 상태)
```
공장 PC : python main.py (통합 런처 = web-api + sequence_service)
현장 관리자 : 공장 PC 화면 또는 AnyDesk로 localhost:8085 접속
```
- ✅ 가장 안전, 구조 단순
- ✅ 로봇·IO 제어 지연 없음 (localhost)
- ❌ 재부팅 시 자동 시작 안 됨 (수동 실행 필요) — **4/14 문제 반복**
- ❌ 태민님 노트북 등 외부에서 Automation 탭 사용 불가

### 🅱️ 시나리오 B: 공장 PC 백엔드 + 원격 UI 접속 (프록시)
```
공장 PC : python main.py (통합 런처)
카카오 VM / 6000 서버 : web-api에 /api/v1/automation/* 를 공장 PC로 프록시
사용자 : 태민님 노트북 → 원격 서버 → (프록시) → 공장 PC
```
- ✅ 태민님 노트북에서도 Automation 탭 동작
- ❌ **원격 로봇 제어 = 안전성 문제** (네트워크 끊김 시 로봇 멈춤 처리 필요)
- ❌ 프록시 코드 추가 구현 + 테스트 필요 (1~2일)
- ❌ 4/23 Basler 도착 코앞 → 다른 리스크 추가 금물

### 🅲 시나리오 C: 원격 접속 시 Automation 탭 가드
```
현장 관리자 (공장 PC localhost) : Automation 탭 활성화
외부 접속자 (태민/예승님 노트북) : Automation 탭 비활성화 + 안내 메시지
```
- ✅ 오조작 방지 (원격에서 실수로 로봇 제어 차단)
- ✅ 구현 간단 (프론트엔드 조건부 렌더링, 30분~1시간)
- ❌ 태민님 노트북 등 외부에서 Automation 탭 사용 여전히 불가 (의도된 제약)

### 🔴 P5. 예승님 안건 — DB 위치 결정 (4/22 저녁 연락) ★ 최우선

**예승님 질문 원문** (4/22 저녁):
> 웹 서버는 카카오 클라우드로 이전됐지만 sequence_service는 현장 PC에 있어야 함. 현재 클라우드가 현장 DB에 못 붙어서 Automation 탭 사용 불가. 두 가지 중 결정 필요:
> 1. DB를 카카오 클라우드로 올리고 현장 PC가 끌어다 쓰게 (구축 빠름, 인터넷 장애 시 로봇 멈춤)
> 2. 현장 방화벽(MySQL 3306)을 오픈하거나 VPN 연결해서 클라우드가 현장 DB를 읽어가는 방식 (보안 세팅 필요, 로봇 가동 안전)

#### 태민 결론: **옵션 2 방향 (공장 DB 유지), 단 Cloudflare Tunnel로**

**제조 현장 원칙**: 공장의 메인 제어는 인터넷에 걸지 않는다. 인터넷은 가끔 끊기는데, 그때마다 로봇이 멈추면 생산 손실이 큼. 예승님도 이미 옵션 1의 단점을 괄호로 명시하셨고, 사실상 옵션 2 쪽으로 기운 제안임.

| 옵션 | 네트워크 단절 시 | 평상시 지연 | 보안 | 구축 난이도 |
|------|----------------|-----------|------|-----------|
| 1. DB 카카오로 이전 | ❌ 로봇 정지 | 50~100ms/쿼리 | 클라우드 보안 그룹만 | 쉬움 |
| 2a. DB 공장, MySQL 3306 직접 오픈 | ✅ 로봇 동작 | 1ms 로컬 | ❌ 3306 공인 노출 위험 | 중 |
| 2b. DB 공장, WireGuard 추가 | ✅ 로봇 동작 | 5~30ms | ✅ 기존 VPN 방식 | 중 (카카오 VM에 WG 클라이언트) |
| **2c. DB 공장, Cloudflare Tunnel** ★ | ✅ 로봇 동작 | 10~30ms | ✅ Zero Trust 인증 | 쉬움 (초대 받으면) |

**2c를 고르는 이유 3가지**:
1. **로봇 안전성** — 인터넷 끊겨도 공장 PC가 로컬 DB로 자립 운영
2. **이미 하려던 일과 합침** — `factory.flickdone.com` Cloudflare Tunnel은 이미 진행 중. MySQL 경로 하나 더 얹는 형태
3. **데이터 주권** — 자동화 로그(레지스터 값, 작업 이력)는 물리 공정 기록이라 현장 PC 로컬이 자연스러움

**3306 포트 직접 오픈(2a)은 절대 비추천** — MySQL을 인터넷에 그대로 노출은 무차별 공격 대상.

#### ★ 더 근본적인 질문 — DB만 원격으로 풀면 자동화 탭이 정말 동작하나?

sequence_service는 DB 외에도 아래 4개에 붙어야 동작함:
- **Modbus TCP** (127.0.0.1:9100) — 로봇 실시간 통신
- **Vision TCP** (127.0.0.1:9200) — 비전 PC 통신
- **Ajin IO** (AXL.dll WinDLL) — Windows + 물리 보드 필수
- **PreFormServer** (44388) — 프린터 제어

이 넷은 **공장 PC 로컬에서만** 동작 가능. 즉 `sequence_service` 자체는 어떤 옵션을 선택해도 **계속 공장 PC에서 실행**되어야 함.

그럼 카카오 VM이 DB를 읽어서 뭘 하려는 건지 확인 필요:

- **(a) UI에서 자동화 이력·상태 조회만** → 옵션 2c로 해결 ✅
- **(b) 카카오 VM에서 "자동화 시작" 버튼 누르면 실제 로봇이 움직이게** → DB 원격만으로는 부족. `sequence_service`에 HTTP API를 열고 웹 API가 그 API를 프록시로 호출해야 함 (시나리오 B, 현재 문서 3.B 참조)

**내일 회의 첫 확인 질문 ①**: 예승님이 생각하는 건 (a)인가 (b)인가?
- (a)면 → DB 원격 접근만 열면 됨 → **옵션 2c**
- (b)면 → sequence_service API 설계부터 논의 → 시나리오 B 병행 + 안전 장치 필요

#### 타이밍 관점 — 내일 합의하고 구현은 Cloudflare 초대 후로

- 대표님 Cloudflare 계정 초대 대기 중 (권한 없음)
- 4/23 Basler 입고 + 리허설 준비가 우선 (다른 리스크 추가 금물)
- 3306 포트는 한 번 뚫으면 되돌리기 어려움 (보안 관성)

**권장 진행**:
| 시점 | 조치 |
|------|------|
| 지금 ~ Cloudflare 초대 | **공장 PC 단독 운영 + Automation 탭은 클라우드에서 비활성 가드** (= 시나리오 A + C) |
| Cloudflare 초대 후 | **옵션 2c 정식 구축** (Tunnel + MySQL Private 접근) |

#### 예승님 답변 초안

> 옵션 2 방향으로 가는 게 맞습니다. 현장 로봇 가동 안전성이 우선이고, 저희도 `factory.flickdone.com` **Cloudflare Tunnel**을 이미 준비 중이라 그 위에 MySQL 프라이빗 접근을 얹는 방식이 자연스럽습니다 (3306 포트를 인터넷에 직접 오픈하는 건 보안상 피하고 싶습니다).
>

1. **현재 공장 PC MySQL 접속 정보** — host, port, user, db명 (예승님이 `sequence_service/.env` 또는 config에서 확인)
2. **DB 스키마 규모** — 자동화 로그 테이블 수, 예상 일일 트래픽 (Tunnel 대역폭 사이징)
